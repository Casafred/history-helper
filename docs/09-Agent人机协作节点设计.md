# PatentLens Agent - 自动化节点与人机确认节点设计

> **补充文档 v1.0**
> 日期: 2026-07-11
> 核心问题：哪些步骤Agent可以全自动走完，哪些节点必须停下来"问人"，确认时如何呈现

---

## 一、现有用户操作流程全景（改造前）

先把用户现在要做的每一步手动操作列清楚，才能判断哪些可以省掉：

```
【一次性配置阶段】
  ① 点击AI设置按钮 → 选择服务商 → 输入API Key → 选模型 → 测试连接 → 保存
  ② （可选）配置OCR引擎

【单次专利分析全流程】
  ③ 在输入框输入专利号
  ④ 点击"查询"按钮 或 按回车
  ⑤ 等待加载（loading转圈）
  ⑥ 自动显示概览Tab（申请信息、当前状态）
  ⑦ 手动点击"同族"Tab查看同族专利
  ⑧ 手动点击"审查文档"Tab
  ⑨ 在文档列表中，手动勾选想要OCR的文档（复选框）
  ⑩ 点击"OCR识别"按钮
  ⑪ 等待每篇文档依次OCR完成（进度不确定）
  ⑫ 手动点击"AI梳理"Tab
  ⑬ （可选）手动选择要分析的文档范围
  ⑭ （可选）调整提示词模板
  ⑮ 点击"开始AI梳理"按钮
  ⑯ 等待流式输出完成（可能1-3分钟）
  ⑰ 阅读分析结果
  ⑱ （可选）点击"导出Word"按钮 → 等待下载
  ⑲ （可选）在PDF阅读器中标注
  ⑳ 如果要查下一个专利：在输入框输入新专利号 → 重复④-⑲
     （注意：最多同时开3个Tab，开第4个时要手动选择关闭哪个）

【批量/特殊场景】
  ㉑ 查询JP日本专利：需单独配置JPO API Key
  ㉒ 查询DE德国专利：只能查注册信息，案卷查阅无法程序化
  ㉓ 引用文献分析：单独点击按钮
  ㉔ 单篇文档深度分析：点击某篇文档旁的"AI分析"按钮
```

**统计：一个完整分析流程，用户需要手动点击 12-18 次，在4个Tab间来回切换，等待多个加载步骤。**

## 二、节点分类：三类自动化程度

根据"是否会造成不可逆后果"、"是否涉及用户主观判断"、"成本是否可控"三个维度，将所有节点分为三类：

| 分类 | 定义 | 处理方式 |
|-----|------|---------|
| 🟢 **绿色节点：全自动执行** | 无副作用、成本极低、结果可预测、不需要主观判断 | 直接执行，只做状态告知，不打断流程 |
| 🟡 **黄色节点：智能决策+事后告知** | Agent可以自行合理决策，但用户可能关心结果 | 执行后在对话中告知"我选择了XXX，因为XXX"，不中断流程；提供"撤销/更改"选项 |
| 🔴 **红色节点：必须先确认再执行** | 涉及不可逆操作、高成本、高风险、强主观偏好 | 暂停流程，弹出明确的确认对话框，用户确认后才继续 |

## 三、🟢 全自动节点（不需要任何确认）

这些步骤Agent可以直接跑完，用户只需要看到进度在动：

| 序号 | 操作 | 为什么可以全自动 | Agent替代后的体验 |
|-----|------|----------------|-----------------|
| 1 | **专利号格式解析与识别** | 纯算法判断，有明确规则，没有歧义 | 自动识别专利局（US/CN/EP/JP等），显示徽章，不需要用户点按钮验证 |
| 2 | **获取专利基本信息**（fetch_patent） | API调用，有缓存，成本为0（你的后端限流已做） | 自动获取申请信息、申请人、当前状态，自动切换到概览Tab |
| 3 | **获取同族专利数据**（fetch_family） | 同一个API调用，无额外成本，数据已在fetch_patent中返回 | 自动拉取同族数据，用户可以随时点击同族Tab查看，不需要手动点 |
| 4 | **获取审查文档列表**（fetch_documents） | 同上，元数据获取无成本 | 自动拿到文档列表，不需要用户等 |
| 5 | **从文档列表中自动筛选关键文档** | 这是一个"规则+启发式"判断，可以全自动：<br>- CTNF/CTFR/CTF → 审查意见类（关键）<br>- 文件名含Response/Reply/Amendment → 答复类（关键）<br>- NTCE/Notice of Allowance → 授权通知<br>- 其他如表单、费用文件 → 非关键 | Agent自动筛选出 2-6 篇核心文档（审查意见+答复对），不需要用户手动勾选。<br>筛选逻辑：<br>① 必须有来有回：有审查意见就要有对应的答复<br>② 轮次优先：第一次审查意见+答复必选，后续轮次视数量决定<br>③ 如果总数 ≤5篇：全部选<br>④ 如果总数 >5篇：选首尾+有实质内容的 |
| 6 | **单篇关键文档下载+OCR**（1-3篇内） | 单篇OCR耗时约5-15秒，免费PaddleOCR无API成本 | 用户说"帮我分析"时，自动下载筛选出的核心文档并OCR，显示进度 |
| 7 | **自动切换Tab显示中间结果** | UI操作，无副作用 | 拿到概览数据切概览Tab，OCR完成切文档Tab显示结果，分析完成切AI梳理Tab |
| 8 | **调用AI进行标准分析**（用默认提示词） | 你现有的默认提示词已经很完善，大多数用户不会改 | 直接用默认的kanbanAnalysis提示词进行完整分析，流式输出 |
| 9 | **缓存已查询过的数据** | 无副作用，提升体验 | 同一个专利号二次提问直接读缓存，告知用户"已从缓存加载" |
| 10 | **API限流/错误自动重试** | 纯技术处理，用户不需要关心 | 遇到429自动等待重试，网络错误自动重试3次，只有最终失败才告知用户 |
| 11 | **从OCR结果中自动提取时间线概要** | 纯结构化处理 | 自动整理文档日期、类型、发文方，作为分析的前置上下文 |
| 12 | **JP/DE专利局自动检测与适配** | 根据专利号前缀自动判断 | 识别到JP专利自动调用JPO客户端，识别到DE自动调用DPMA客户端，不需要用户手动切换模式 |

### 🟢 节点的呈现方式：

这些节点执行时，**不弹对话框、不问问题**，只通过以下方式让用户知道在做什么：

1. **TodoList状态更新**：从 ⏳ → 🔄 → ✅
2. **进度提示文字**：在思考面板或状态栏显示"正在下载第2/4篇文档..."
3. **Tab自动切换**：数据到了就切过去展示，用户能看到结果在"流进来"
4. **非侵入式通知**：右下角toast提示，如"已识别为美国专利"，3秒后自动消失

## 四、🟡 智能决策节点（Agent自己选，但事后告知）

这些节点Agent有合理的默认选择，但选择不同结果会有差异，需要让用户知道你选了什么、为什么这么选，并给一个"反悔"的入口。

| 序号 | 决策点 | Agent的默认选择逻辑 | 事后告知方式 | 用户如何反悔/更改 |
|-----|-------|-------------------|------------|----------------|
| 1 | **OCR引擎选择** | 默认用PaddleOCR（免费、本地、无Key）；如果用户历史上用过GLM OCR，或者文档图片质量差Paddle识别效果不好时，自动切换GLM OCR | 思考面板中显示："默认使用PaddleOCR进行识别（免费）"，如果切换了引擎说明原因 | 在OCR开始前的短暂准备期，显示一个小提示条："使用PaddleOCR [切换为GLM OCR]"，点击即可切换 |
| 2 | **OCR文档数量选择** | 默认选择关键审查意见+答复（2-4篇），不全量OCR | OCR开始前在聊天中显示："找到12篇文档，将自动OCR其中4篇核心文档（2轮审查意见+答复）" | 旁边放两个按钮："全量OCR(12篇)"、"手动选择"，点击后调整选择再继续 |
| 3 | **分析深度/类型选择** | 默认用full_analysis（完整分析报告）；如果用户说"简单看看"用quick_summary；用户说"重点看XXX"则聚焦对应部分 | 分析开始时显示："将执行完整审查历史分析，包含时间线、争议焦点、权利要求演变、无效机会" | 提示条提供选项：[完整分析] [快速摘要] [仅争议焦点] [仅权利要求演变]，点击切换 |
| 4 | **新专利打开Tab时挤掉哪个旧Tab** | 优先关闭：没有未保存分析、没有PDF标注、最早打开的Tab；如果所有Tab都有未保存内容，则升级为🔴确认节点 | 关闭旧Tab时toast通知："已自动关闭最早的USxxx标签页" | Toast上带"撤销"按钮，5秒内点击可恢复 |
| 5 | **引用文献是否自动分析** | 默认分析完主审查流程后，不主动跑引用文献分析（因为增加额外时间），但在最终报告中提示"可以进一步分析引用文献" | 最终回答末尾提示："💡 如需进一步分析审查员引用的X篇对比文献，可以告诉我" | 不需要弹窗，用户看到提示后直接说"帮我分析引用文献"即可 |
| 6 | **分析结果展示位置** | 默认在AI梳理Tab渲染完整Markdown报告 | 自动切换到AI梳理Tab，流式渲染 | 用户可以自由切到其他Tab，后台继续渲染不中断 |
| 7 | **导出文件默认路径和文件名** | 默认路径：`~/PatentLens/Reports/`；默认文件名：`{专利号}_审查分析_{日期}.docx` | 导出成功后显示："已导出到 ~/PatentLens/Reports/US14412875_审查分析_20260711.docx" | 旁边有[更改路径] [打开所在文件夹]按钮 |
| 8 | **错误重试策略** | 网络超时/5xx错误自动重试3次（指数退避）；4xx错误（如API Key错误）不重试，直接报告问题 | 重试时显示："网络超时，正在重试 (2/3)..." | 如果重试2次仍失败，升级为🔴节点，问用户是继续重试还是跳过 |

### 🟡 节点的呈现方式：

关键设计原则：**先执行，不阻塞流程，但让选择可见可改**

呈现形态：**轻量级提示条（Inline Tip/Banner）**，嵌入在对话流中或进度区，不是模态弹窗：

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 找到12篇审查文档，将自动OCR其中4篇核心文档                 │
│   （第1/2次审查意见 + 答复） 预计耗时约2分钟                  │
│                                                             │
│    [✅ 确认，开始OCR]  [📋 手动选择文档]  [📄 全量OCR(12篇)] │
└─────────────────────────────────────────────────────────────┘
```

特点：
- 不使用模态对话框（不阻断）
- 默认选项高亮（用户什么都不做也会继续）
- 给了3-5秒窗口可以点其他选项，过了窗口自动继续
- 提示条会保留在对话历史中，用户事后想调整可以从这里重新发起

## 五、🔴 必须确认节点（不确认不能继续）

这些节点会产生不可逆后果、消耗显著资源、或涉及强主观偏好，**必须停下来等用户明确同意才继续**：

| 序号 | 确认节点 | 为什么必须确认 | 确认时机 | 默认选项（如有） |
|-----|---------|--------------|---------|----------------|
| 1 | **批量下载+OCR超过5篇文档** | 每篇OCR约5-15秒，5篇以上可能需要1-2分钟；如果用GLM OCR还会消耗API额度；用户可能等不了那么久，或者只想看关键的 | Agent筛选完文档发现≥5篇时，在OCR开始前 | 取消（不默认执行，需要用户主动确认） |
| 2 | **首次使用/AI API Key未配置** | 没有API Key根本无法进行AI分析，这是前置条件 | 用户第一次说"帮我分析"且检测到没有配置Key时 | 打开设置面板 |
| 3 | **OCR引擎为GLM且批量处理≥3篇** | GLM OCR是按调用计费的，批量会产生费用 | 选择GLM OCR且文档≥3篇时 | 切换到PaddleOCR |
| 4 | **导出文件覆盖已存在文件** | 文件覆盖是不可逆操作 | 导出前检测到同名文件已存在时 | 自动重命名（如`US14412875_审查分析_20260711 (1).docx`） |
| 5 | **关闭有未保存内容的Tab** | 用户可能花了10分钟看的分析结果没了，PDF标注丢了 | 新专利查询需要挤掉旧Tab，而旧Tab有未保存分析/标注时 | 取消（不关闭） |
| 6 | **批量处理≥3个专利** | 多专利处理可能耗时10分钟以上，消耗大量API配额 | 用户输入3个以上专利号要求批量分析时 | 取消 |
| 7 | **需要用户提供主观判断/选择方向** | 有些问题没有"正确答案"，纯看用户目的是什么。例如：<br>- "帮我分析这个专利"但没说分析什么方面（无效？FTO？授权前景？）<br>- 有多个可能的专利号匹配（用户输入"14412875"没带国家代码，可能是US/EP/CN等） | Agent理解完需求但存在关键歧义时 | 给出选项列表让用户选 |
| 8 | **工具连续失败3次以上** | 可能是配置错误、网络问题、API挂了，Agent自己解决不了，需要用户介入 | 连续3次重试失败后 | 中止流程，报告错误 |
| 9 | **用户要求停止/中断当前任务** | 用户随时可以说"停"、"算了"、"取消" | 用户发出中断指令时 | 立即停止 |
| 10 | **JPO API Key未配置但要查日本专利** | JPO需要单独配置Key，和通用AI Key是分开的 | 识别到JP专利但检测到没配置JPO Key时 | 提示用户配置 |
| 11 | **在现有分析结果上重新分析会覆盖** | 用户已经跑过一次AI梳理，如果重新分析会替换现有结果 | 用户说"重新分析"但检测到已有分析结果时 | 在现有结果基础上追加（不覆盖） |

## 六、🔴 确认节点的UI呈现设计

确认不是简单弹一个`confirm()`对话框，要根据风险等级和场景使用不同的呈现方式。

### 类型1：选项提问式（适用于方向选择、歧义澄清）

**场景**：用户输入"14412875"没带国家代码 / 用户说"帮我分析这个专利"没说分析方向

**呈现**：在聊天流中直接渲染选择题（类似TRAE的AskUserQuestion），用户点一下就能继续：

```
┌──────────────────────────────────────────────────────────────┐
│ 🤔 请确认一下我理解对了吗？                                    │
│                                                              │
│ 你输入了 "14412875"，可能对应以下多个专利，请选择要查询的：    │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ ○ US14412875B2  美国专利  已授权  2015年授权             │ │
│ │ ○ EP14412875B1  欧洲专利  已授权  2017年授权             │ │
│ │ ○ CN201410412875.X  中国专利  审查中                     │ │
│ └──────────────────────────────────────────────────────────┘ │
│                                                              │
│ 另外，你希望我从哪个角度分析？（可多选）                       │
│ ☑ 完整审查历史分析（时间线+争议焦点+权利要求演变）            │
│ ☐ 快速状态摘要（5分钟出结果）                                 │
│ ☐ 重点看创造性争议部分                                        │
│ ☐ 重点看权利要求修改和禁反言                                  │
│                                                              │
│                   [ 确认并开始分析 ]                          │
└──────────────────────────────────────────────────────────────┘
```

**交互要点**：
- 问题明确，不要开放式提问
- 提供预置选项，用户点一下就行，不用打字
- 默认选项预勾选（根据上下文猜测最可能的意图）
- 一个对话框最多问1-2个问题，不要一次问5个
- 底部明确的行动按钮

---

### 类型2：成本/风险确认式（适用于批量操作、可能产生费用）

**场景**：批量OCR超过5篇 / 批量分析3个以上专利 / 使用付费OCR引擎

**呈现**：模态确认框，明确告知**做什么、多久、会产生什么**：

```
┌──────────────────────────────────────────────────────────────┐
│  ⚠️  即将执行批量操作，请确认                                  │
│  ────────────────────────────────────────────────────────    │
│                                                              │
│  我找到了这个专利包的 18 篇审查文档，你选择了全量OCR。         │
│                                                              │
│  📊 预估：                                                   │
│     • 文档数量：18篇（含4次审查意见+6次答复+8篇其他）         │
│     • 预计耗时：8-15分钟                                     │
│     • OCR引擎：PaddleOCR（免费本地引擎）                      │
│     • AI分析：分析Token预估约 40K-60K tokens                  │
│                                                              │
│  💡 建议：核心的审查意见和答复只有前两轮共6篇，分析这6篇       │
│     就能覆盖90%的关键信息，耗时约3分钟。                       │
│                                                              │
│                                                              │
│  [ 🔄 只分析核心6篇（推荐） ]  [ ✅ 确认全量OCR 18篇 ]        │
│  [ ❌ 取消 ]                                                │
└──────────────────────────────────────────────────────────────┘
```

**交互要点**：
- 用⚠️图标提示这是一个需要注意的操作
- 给出具体数字：多少篇、多久、大概多少token/费用（如果用付费API）
- 主动给出"推荐方案"（更节省时间/成本的选项），不把用户抛在那里自己想
- 按钮有主次：推荐选项是主按钮，全量是次按钮，取消是文字按钮
- 如果用付费API（GLM OCR/OpenAI），要明确标"预计费用：约¥0.xx"

---

### 类型3：文件覆盖/不可逆操作确认（高风险）

**场景**：覆盖已有文件 / 关闭有未保存内容的Tab

**呈现**：强确认对话框，明确告知"会失去什么"：

```
┌──────────────────────────────────────────────────────────────┐
│  🛑 文件已存在，是否覆盖？                                    │
│  ────────────────────────────────────────────────────────    │
│                                                              │
│  路径：~/PatentLens/Reports/US14412875_审查分析.docx           │
│  修改时间：2026-07-10 15:32（昨天）                           │
│                                                              │
│  覆盖后，原文件内容将无法恢复。                                │
│                                                              │
│                                                              │
│  [ 📝 自动重命名（推荐） ]  [ ✅ 覆盖原文件 ]  [ ❌ 取消 ]   │
│                                                              │
│  （自动重命名会保存为：US14412875_审查分析 (1).docx）          │
└──────────────────────────────────────────────────────────────┘
```

**交互要点**：
- 🛑红色/高对比图标提示风险等级
- 明确告知不可逆后果："覆盖后无法恢复"
- **默认选项永远不是"覆盖"**，默认是自动重命名或者取消
- 提供最安全的选项作为"推荐"
- 给出具体的文件信息（路径、时间）让用户确认是哪个文件

## 七、智能默认：让Agent"懂你"而不是每次都问

为了减少确认次数，Agent要记住用户的偏好和历史行为，形成"智能默认"：

| 偏好项 | 学习方式 | 默认值 |
|-------|---------|-------|
| 喜欢的分析深度 | 记住用户连续3次选择"完整分析"还是"快速摘要" | 完整分析 |
| 默认OCR引擎 | 记住用户上次用的什么 | PaddleOCR |
| 是否自动导出 | 记住用户是否每次都导出 | 询问前3次，之后记住选择 |
| 导出格式 | 记住用户最常用docx还是pdf | docx |
| 导出路径 | 记住上次导出到哪个文件夹 | ~/PatentLens/Reports/ |
| 核心文档数量偏好 | 记住用户通常选4篇还是全量 | 4篇核心 |
| 是否自动打开导出文件 | 记住用户习惯 | 不打开 |

**实现方式**：存在`localStorage`里，简单键值对即可，不需要复杂的用户模型。

## 九、确认频率控制原则：不要变成"提示框地狱"

最后也是最重要的一条原则：**Agent的目标是减少用户操作，不是把点击从按钮变成确认框**。

严格遵循以下规则：

1. **同一流程中🔴确认弹窗不超过2次**。如果超过了，说明你的默认选择逻辑没做好，回去优化Agent的智能默认。
2. **能🟡事后告知的就🟡，不要🔴**。只有真正不可逆/高成本的才弹模态框。
3. **能🟢自动处理的连告知都不用太详细**。进度条和TodoList动就可以了，不用每一步都在聊天里发一条消息。
4. **记住用户选择，下次不重复问**。第一次问"导出后自动打开吗？"，记住答案，后面都不这么问了。
5. **3秒规则**：🟡的提示条如果3-5秒用户没反应，就按默认选项继续，不要一直等着。
6. **用户说"以后都这样"就记住**。如果用户在确认时说了"以后都按这个来"或者勾选了"记住我的选择"，就存为偏好。

**理想体验目标**：
- **简单请求**（"查一下USxxx现在什么状态"）：零次确认，一句话等结果
- **标准请求**（"帮我分析USxxx，导出Word"）：零次或一次🟡提示，无弹窗
- **复杂/歧义请求**（输入不明确+批量操作）：最多两次🔴确认
- **配置缺失/错误**：一次🔴确认+一键修复按钮
