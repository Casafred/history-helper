# 01 - 项目概述与技术选型

## 1. 项目背景

在专利代理实务中，梳理专利审查历史是一项高频但耗时的重复性工作。代理人需要：

- 将不同地区的专利号格式进行转换（如中国申请号 → 美国申请号格式）
- 逐一登录各国专利局网站查询审查历史
- 下载并阅读审查意见通知书（Office Action）
- 整理审查时间线、驳回理由、引用文献等关键信息

当前痛点：
1. **多平台切换**：美国、中国、欧洲、日本、韩国各局系统独立，操作流程各异
2. **格式转换繁琐**：不同国家专利号格式不同，手动转换容易出错
3. **审查信息分散**：同族专利在不同局的审查进度需要分别查询
4. **梳理效率低**：审查历史文档量大，人工梳理耗时

## 2. 需求分析

### 2.1 核心需求

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 专利号格式转换 | P0 | 支持中美欧日韩五局专利号格式互转 |
| 美国专利审查历史获取 | P0 | 通过 USPTO ODP API 获取完整审查历史 |
| 同族专利审查信息获取 | P1 | 通过 Global Dossier 获取 IP5 同族信息 |
| 审查历史时间线展示 | P0 | 可视化展示审查流程和关键节点 |
| AI 辅助内容梳理 | P2 | 接入 AI 对审查意见进行摘要和梳理 |

### 2.2 非功能需求

| 需求 | 目标 |
|------|------|
| 体积 | 打包后 10MB 以内 |
| 平台 | 主要支持 Windows，开发环境为 Windows |
| 交叉编译 | 支持 Linux 下交叉编译生成 Windows 可执行程序 |
| 并发 | 低请求量（单人使用），无需高并发设计 |
| 部署 | 零依赖，单文件可执行，无需安装 Python 等运行时 |
| 安全 | API 密钥本地存储，禁止泄露至远程仓库 |

## 3. 技术选型论证

### 3.1 开发语言：Rust

**选择 Rust 的理由：**

| 维度 | Rust | Python | Go | C++ |
|------|------|--------|-----|-----|
| 编译体积 | ✅ 极小（静态链接后 <10MB） | ❌ 需 Python 运行时 | ✅ 较小 | ✅ 较小 |
| 零依赖部署 | ✅ 单文件可执行 | ❌ 需安装解释器 | ✅ 单文件 | ⚠️ 可能缺 DLL |
| 内存安全 | ✅ 编译期保证 | ✅ GC 管理 | ✅ GC 管理 | ❌ 手动管理 |
| HTTP 生态 | ✅ reqwest 成熟 | ✅ requests 成熟 | ✅ net/http 内置 | ⚠️ 需第三方 |
| 桌面框架 | ✅ Tauri 原生支持 | ⚠️ PyQt/tkinter 体积大 | ⚠️ Wails/Fyne | ⚠️ Qt 体积大 |
| 交叉编译 | ✅ rustup + cross | ❌ 复杂 | ✅ 内置支持 | ⚠️ 工具链复杂 |

**结论**：Rust 在体积控制、零依赖部署、内存安全、桌面框架支持等方面全面优于其他选项。

### 3.2 桌面框架：Tauri

**选择 Tauri 的理由：**

- **体积优势**：Tauri 使用系统 WebView，不打包 Chromium，最终体积可控制在 3-8MB
- **Rust 原生**：后端逻辑用 Rust 编写，前端用 HTML/CSS/JS
- **安全模型**：默认最小权限，API 调用需显式声明
- **跨平台**：支持 Windows / macOS / Linux

对比 Electron：
- Electron 打包后通常 100MB+（内含 Chromium + Node.js）
- Tauri 打包后通常 3-8MB（使用系统 WebView）

### 3.3 HTTP 客户端：reqwest

- Rust 生态中最成熟的 HTTP 客户端
- 支持 async/await、TLS、代理、超时
- 支持 JSON 序列化/反序列化

### 3.4 HTML 解析：selectors

- 用于解析 Global Dossier 等无公开 API 的页面
- 轻量级 CSS 选择器引擎

## 4. 数据来源

### 4.1 USPTO ODP API（主要数据源）

- **地址**：https://api.uspto.gov
- **文档**：https://data.uspto.gov/swagger/index.html
- **认证**：API Key（X-API-KEY 请求头）
- **费用**：免费
- **覆盖范围**：美国专利申请全量数据
- **数据内容**：
  - 申请元数据（申请人、发明人、分类号等）
  - 审查历史事件（Transaction History）
  - 审查文档（Office Action 全文 PDF）
  - 续案/分案关系（Continuity Data）
  - 专利权转让记录（Assignment）
  - 外国优先权（Foreign Priority）

### 4.2 USPTO Office Action API（补充数据源，迁移中）

- **Office Action Text Retrieval API**：审查意见全文检索
- **Office Action Rejection API**：驳回理由结构化数据
- **Office Action Citations API**：引用文献数据
- **注意**：这三个 API 将于 2026 年 5 月 29 日迁移至 ODP

### 4.3 Global Dossier（同族专利数据源）

- **地址**：https://globaldossier.uspto.gov
- **性质**：Web 应用，无公开 REST API
- **覆盖范围**：IP5 局（美国、欧洲、日本、韩国、中国）+ WIPO CASE 附加国家
- **数据内容**：
  - 同族专利列表
  - 各局审查状态
  - 审查意见文档下载
  - 机器翻译
- **接入方式**：需通过 HTTP 请求模拟浏览器行为，解析返回的 HTML/JSON

### 4.4 数据来源对比

| 维度 | USPTO ODP API | Global Dossier |
|------|---------------|----------------|
| 数据范围 | 仅美国 | IP5 五局 + WIPO |
| 接入方式 | 标准 REST API | Web 应用（需模拟） |
| 认证 | API Key | 无需（公开访问） |
| 稳定性 | ✅ 高（官方 API） | ⚠️ 中（页面可能改版） |
| 数据结构 | ✅ 结构化 JSON | ❌ 需解析 |
| 审查意见全文 | ✅ 可下载 PDF | ✅ 可下载 PDF |
| 同族信息 | ⚠️ 仅美国续案关系 | ✅ 跨局同族 |
