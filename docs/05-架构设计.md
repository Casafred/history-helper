## 2. 数据流

### 2.1 查询审查历史（主流程）

```
用户输入专利号 → 前端 → Tauri IPC → Rust 后端 → USPTO API → 返回数据 → 前端渲染
```

详细步骤：

```
1. 用户在搜索框输入 "US14412875"，按回车或点击"查询"
2. 前端调用 invoke("detect_patent_office", { input: "US14412875" })
   → 后端 patent::converter::detect_office() 识别为 US
   → 前端显示 "US 专利" 徽章
3. 前端调用 invoke("fetch_application", { appNumber: "US14412875" })
   → 后端 patent::converter::normalize_us_application_number() 转为 "14412875"
   → 后端 api::uspto::UsptoClient::get_application("14412875")
   → reqwest GET https://api.uspto.gov/api/v1/patent/applications/14412875
     Header: X-API-KEY: <from .env>
   → 返回 PatentApplicationResponse JSON
   → 后端包装为 CommandResult { success: true, data: ... }
   → 前端 renderOverview() 渲染概览面板
4. 前端依次调用 fetch_transactions / fetch_documents / fetch_continuity
   → 每次调用间由后端 1.5s 间隔限流保护
   → 前端分别渲染时间线、文档列表、续案关系
```

### 2.2 专利号格式转换

```
用户输入专利号 → 前端 → Tauri IPC → patent::converter::parse_patent_number()
→ 识别专利局 + 标准化格式 → 返回 PatentNumber 结构 → 前端渲染
```

### 2.3 审查文档下载

```
前端获取文档列表 → 提取 downloadUrl → 用户点击 PDF 链接
→ 浏览器直接打开下载 URL（需带 X-API-KEY，当前由 USPTO CDN 处理）
```

> 注意：当前文档下载链接由 USPTO API 返回，部分链接可能需要 API Key 认证。
> 后续版本可能需要通过 Rust 后端代理下载。

## 3. 模块职责

### 3.1 api/uspto.rs — USPTO API 客户端

| 组件 | 职责 |
|------|------|
| `UsptoClient` | 封装 reqwest Client + API Key，提供所有 USPTO API 调用方法 |
| `UsptoApiError` | 统一错误类型（MissingApiKey / RequestFailed / RateLimited / NotFound / ApiError） |
| `get_with_retry()` | 核心请求方法，实现指数退避重试（最多 3 次） |

**限流策略**：
- 每次请求前等待 1.5 秒（`REQUEST_INTERVAL_MS`）
- 遇 429 状态码时，指数退避：1.5s → 3s → 6s
- 单次请求超时 30 秒

**API Key 管理**：
- 从环境变量 `USPTO_API_KEY` 读取（通过 `dotenv` 加载 `.env` 文件）
- 通过 `X-API-KEY` 请求头传递
- 客户端实例通过 Tauri State 懒初始化并复用

### 3.2 patent/converter.rs — 专利号格式转换

| 函数 | 职责 |
|------|------|
| `detect_office()` | 根据前缀识别专利局（US/CN/EP/JP/KR/WO） |
| `normalize_us_application_number()` | 美国申请号标准化（去前缀、8位数字格式） |
| `normalize_cn_application_number()` | 中国申请号标准化（去点号、统一前缀） |
| `parse_patent_number()` | 完整解析：识别 + 标准化，返回 PatentNumber 结构 |

**识别规则**：
- `US` 前缀或 8 位纯数字开头 → US
- `CN` 前缀 → CN
- `EP` 前缀 → EP
- `JP` 前缀 → JP
- `KR` 前缀 → KR
- `WO` 或 `PCT` 前缀 → WO

### 3.3 parser/office_action.rs — 审查意见解析

| 组件 | 职责 |
|------|------|
| `OfficeActionType` | 审查意见类型枚举（Non-Final / Final / Advisory 等） |
| `EventCategory` | 事件分类（OfficeAction / ApplicantResponse / FeePayment / Publication / Other） |
| `parse_office_actions()` | 从文档列表中筛选审查意见类文档 |
| `build_timeline()` | 构建完整审查时间线（事件 + 审查意见） |

**文档代码映射**：
- `CTNF` → Non-Final Rejection
- `CTF` → Final Rejection
- `CTFR` → Advisory Action
- `REST` → Restriction Requirement
- `NTCE` → Notice of Allowance
- `EX.Q` → Ex Parte Quayle

### 3.4 models/patent.rs — 数据模型

面向业务的高层数据模型，用于后续 AI 梳理等功能：

| 结构体 | 用途 |
|--------|------|
| `PatentSearchResult` | 搜索结果摘要 |
| `ExaminationHistory` | 完整审查历史（时间线 + 审查意见 + 续案关系） |
| `OfficeActionSummary` | 审查意见摘要 |
| `ContinuityInfo` | 续案/分案关系 |
| `ForeignPriorityInfo` | 外国优先权信息 |

### 3.5 lib.rs — Tauri 命令层

| 命令 | 对应前端操作 | 调用的模块 |
|------|-------------|-----------|
| `convert_patent_number` | 格式转换按钮 | patent::converter |
| `detect_patent_office` | 输入框实时识别 | patent::converter |
| `fetch_application` | 查询按钮（申请数据） | api::uspto |
| `fetch_transactions` | 查询按钮（审查事件） | api::uspto |
| `fetch_documents` | 查询按钮（文档列表） | api::uspto |
| `fetch_continuity` | 查询按钮（续案关系） | api::uspto |
| `fetch_foreign_priority` | 查询按钮（外国优先权） | api::uspto |

## 4. 前后端通信机制

### 4.1 Tauri IPC 调用方式

前端通过 `window.__TAURI__.core.invoke()` 调用后端命令：

```javascript
const result = await invoke("fetch_application", { appNumber: "14412875" });
```

### 4.2 统一返回格式

所有命令返回统一的 `CommandResult` 结构：

```json
{
  "success": true,
  "data": { ... },
  "error": null
}
```

或错误时：

```json
{
  "success": false,
  "data": null,
  "error": "API key not configured. Set USPTO_API_KEY in .env file"
}
```

### 4.3 前端字段映射

前端直接使用 USPTO API 返回的 camelCase 字段名（如 `applicationNumberText`、`eventDataBag`），
与 Rust 后端的 serde 反序列化字段一一对应。详见 [API 数据模型与字段映射](07-API数据模型与字段映射.md)。

## 5. 状态管理

### 5.1 后端状态

通过 Tauri State 管理 `UsptoClient` 实例：

```rust
struct AppState {
    uspto_client: Mutex<Option<UsptoClient>>,
}
```

- **懒初始化**：首次 API 调用时创建 UsptoClient，后续复用
- **线程安全**：使用 `Mutex` 保护，支持多线程访问
- **API Key 读取**：创建时从环境变量读取，运行时不变

### 5.2 前端状态

当前前端无全局状态管理，数据通过 DOM 直接渲染。后续如需复杂状态可引入轻量状态管理。

## 6. 安全模型

| 安全措施 | 实现方式 |
|----------|----------|
| API Key 保护 | .env 文件 + .gitignore，禁止提交至远程仓库 |
| 日志脱敏 | 日志中禁止输出完整 API Key |
| CSP 策略 | tauri.conf.json 中 `security.csp: null`（开发阶段，后续收紧） |
| 请求标识 | User-Agent: `PatentHistoryHelper/0.1.0` |
| 数据使用 | 专利文档仅限内部梳理，禁止对外传播或商用 |

## 7. 后续扩展点

### 7.1 Global Dossier 接入

需新增 `api/global_dossier.rs` 模块，通过 HTTP 请求模拟浏览器行为获取同族专利数据。
Global Dossier 无公开 REST API，需解析 HTML/JSON 响应。

### 7.2 AI 梳理接入

需新增 `ai/` 模块，接入 LLM API（如 OpenAI / 本地模型），对审查意见进行摘要分析。
需设计 prompt 模板和结果解析逻辑。

### 7.3 文档代理下载

当前文档下载链接直接由前端打开，部分链接可能需要 API Key 认证。
后续需在 Rust 后端实现代理下载，通过后端添加 X-API-KEY 头后转发文件流。

### 7.4 本地缓存

当前无本地缓存，每次查询都发起 API 请求。
后续可引入 SQLite 或文件缓存，避免重复查询同一专利。
