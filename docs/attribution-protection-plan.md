# PatentLens 代码身份标注与防盗方案

> 主贡献者：Alfred Shi
> 适用项目：PatentLens（Tauri/Electron + 本地 Node Server 专利审查工具）
> 文档版本：v1.0 · 2026-07-01

---

## 1. 背景与目标

### 1.1 目标
1. **身份清晰**：任何人在使用、查看、二次分发 PatentLens 时，都能明确知道 **Alfred Shi 是主要贡献者**。
2. **提高挪用门槛**：使"直接拷贝前端代码即可独立运行"变得困难。
3. **不影响正常使用与维护**：防盗措施不得显著增加正版用户的负担或拖慢开发迭代。

### 1.2 现实约束（必须先认清）
- PatentLens 是 **客户端应用**：前端 JS（`src/scripts/*.js`）会被发送到用户机器并执行，**任何纯前端的"防盗"在技术上都可被逆向**。
- 混淆/压缩只是 **抬高门槛**，不是加密；对决心逆向的人无效。
- 真正有效的保护 = **核心逻辑不下发到前端** + **身份标注永久可见** + **轻量混淆增加溯源成本**。
- 对团队内部工具而言，**强防盗（加壳/VMP/机器码）性价比低**，且会损害正版体验。

因此本方案的定位是：**务实可控，标注为主、防盗为辅**。

---

## 2. 方案分层

| 层级 | 名称 | 成本 | 效果 | 推荐度 |
|------|------|------|------|--------|
| A | 身份标注 | 零 | 让所有人看到 Alfred Shi 是主贡献者 | ★★★★★ 必做 |
| B1 | 构建期混淆 | 中 | 前端 JS 难读、难维护 | ★★★★ 推荐 |
| B2 | 关键逻辑后移 | 中 | 拿走前端也跑不起来 | ★★★★ 推荐 |
| B3 | License/机器码 | 高 | 限制未授权使用 | ★★ 视场景 |
| C | 强防盗（加壳/VMP） | 高 | 有限收益 | ★ 不推荐 |

**推荐组合：A 全套 + B1 + B2**。

> 若项目无 `package.json`，建议补一个最小版本用于元数据沉淀。

### 3.2 LICENSE 文件
根目录新增 `LICENSE`，建议采用 **MIT** 或 **GPL-3.0**：
- **MIT**：允许商用但要求保留版权声明（Alfred Shi 永久署名）
- **GPL-3.0**：强制衍生作品同样开源，更适合"不希望被闭源挪用"的场景

推荐 GPL-3.0，头部固定写入：
```

Copyright (C) 2026  Alfred Shi

This program is free software: you can redistribute it and/or modify
it under the terms of the GNU General Public License as published by
the Free Software Foundation, either version 3 of the License, or
(at your option) any later version.

```

### 3.3 源文件头注释（每个 .js / .css / .html 顶部）
统一格式：
```js
/*!
 * PatentLens - 专利审查文档获取与梳理
 * Copyright (C) 2026 Alfred Shi. All rights reserved.
 * @author      Alfred Shi
 * @license     GPL-3.0

```
CSS / HTML 同理。**关键点**：注释里的 `@author` 在混淆后仍会保留（terser 默认保留带 `!` 的版权注释）。

### 3.4 App UI 永久署名（用户可见层）
1. **启动页（splash-screen）**：在 Logo 下方增加一行 `Developed by Alfred Shi`。
2. **页脚/关于对话框**：增加"关于 PatentLens"入口，弹出框含作者、版本、许可证、源地址。
3. **设置面板**：底部固定一行小字 `© 2026 Alfred Shi · GPL-3.0`。
4. **导出文件水印**：导出的 Word/PDF 在末尾或页脚注入 `Generated by PatentLens (Alfred Shi)` —— 既标注身份，又可溯源。

### 3.5 Git 层署名
- `.mailmap` 统一 Alfred Shi 的多邮箱/多用户名为一条记录，`git shortlog` 时贡献清晰。
- 后续提交统一用 Alfred Shi 的 name/email（不在此自动改 git config，需用户确认）。
- 可加 `AUTHORS` 文件列出贡献者，Alfred Shi 排首位并标注 `maintainer`。

## 4. B1 层 — 构建期混淆与压缩（推荐）

### 4.1 工具选型
| 工具 | 用途 | 备注 |
|------|------|------|
| **terser** | 压缩 + 变量重命名 | 轻量，必用 |
| **javascript-obfuscator** | 字符串数组化、控制流平坦化、防格式化 | 防读性强 |
| **cssnano** | CSS 压缩 | 顺带 |

### 4.2 流程
1. 新增 `build/obfuscate.js` 脚本：
   - 读取 `src/scripts/*.js`
   - terser 压缩 → javascript-obfuscator 混淆
   - 输出到 `dist/scripts/*.js`
   - 保留带 `!` 的版权注释头
2. `package.json` 增加 `scripts.build`：`node build/obfuscate.js`
3. **发布模式**：server.js / electron-main.js 在生产环境从 `dist/` 提供静态资源；开发模式仍用 `src/`。
4. 通过环境变量 `NODE_ENV=production` 切换。

### 4.3 混淆参数建议（平衡可读性破坏与性能）
```js
{
  compact: true,
  controlFlowFlattening: true,        // 控制流平坦化（中等强度）
  controlFlowFlatteningThreshold: 0.5,
  stringArray: true,                  // 字符串数组化
  stringArrayEncoding: ['base64'],
  stringArrayThreshold: 0.75,
  identifierNamesGenerator: 'mangled',
  // 保留版权注释
  sourceMap: false,                   // 发布版不输出 sourcemap
}
```
> 注意：`controlFlowFlattening` 会带来 10%~30% 性能损耗，对本应用（IO 为主）可接受。

### 4.4 限制与风险
- 混淆后 bug 排查变难 → **开发模式绝不混淆**，仅发布混淆。
- 仍可被决心者反混淆，所以必须配合 B2（逻辑后移）才有意义。

## 5. B2 层 — 关键逻辑后移（推荐，最有效）

### 5.1 原则
**"前端只负责展示与交互，所有不可替代的业务逻辑留在 server.js / electron-main.js"**——这两个文件随 Electron 二进制分发，逆向难度远高于明文 JS。

### 5.2 现状盘点与迁移清单
| 逻辑 | 当前位置 | 是否核心 | 建议位置 |
|------|----------|----------|----------|
| Google Patents HTML 解析 | `patent-parser.js`（共享） | ★核心 | 已被 server.js/electron-main.js 引用 ✓ 保留 |
| 专利号规范化 | `patent-parser.js` | ★核心 | 保留在后端 ✓ |
| AI PROMPT 模板 | `src/scripts/web-ai.js` | ★核心 | **迁移到后端**，前端只传 key |
| AI streamChat 请求构造 | `src/scripts/web-ai.js` | 中 | 建议改走后端代理（同时隐藏 API Key） |
| 审查文档 PDF URL 构造 | `src/scripts/web-app.js` | 中 | 迁移到后端，前端只拿现成 URL |
| OCR / 翻译调用 | 前端直连 | 低 | 视情况 |
| 看板分类、时间线生成 | 前端 | 低 | 可留前端 |

### 5.3 具体动作
1. **AI PROMPT 后移**：`DEFAULT_PROMPTS`（含 `patentInterpretation` 等）从 `web-ai.js` 移到 `server.js`，前端 `streamChat` 改为调用 `/api/ai/chat` 代理接口。这样：
   - PROMPT 不下发到前端（防抄）
   - API Key 不暴露在前端（安全收益更大）
2. **AI 请求代理**：新增 `/api/ai/stream` SSE 接口，后端转发到 OpenAI/DeepSeek/智谱，前端只消费流。
3. **PDF URL 后端签发**：前端不再拼 `/api/gd/doc-content/...`，改由后端返回签名 URL。

### 5.4 收益
- 拿走前端 JS 也无法独立运行（无 PROMPT、无 AI 代理、无 PDF URL 签发）。
- 顺带修复了"API Key 写在前端"的潜在安全问题。

## 6. B3 层 — License / 机器码校验（视场景，谨慎）

### 6.1 方案
- 启动时读取本地 `license.dat`（含授权机器指纹 + 签名）。
- 校验：机器码（MAC/CPU ID 哈希）匹配 → 放行；不匹配 → 限制查询次数/禁用导出。
- License 由 Alfred Shi 私钥离线签发。

### 6.2 风险
- 增加分发与维护成本（每个用户要签发 license）。
- 私钥泄露则体系崩溃。
- 对内部团队工具，**通常不需要**。

**建议**：除非将来要对外分发收费，否则不做。若做，仅在 Electron 主进程校验（前端校验易被绕过）。

- 对本项目无必要。

## 10. 维护与风险提示
1. **混淆 vs 调试**：务必保留"开发模式不混淆"开关，线上 bug 需能在本地复现。
2. **版权注释保留**：terser 配置 `format.comments: /^!/`，确保 `/*! ... */` 头不被剥离。
3. **GPL 传染性**：选 GPL-3.0 意味着衍生作品必须同样 GPL 开源——若希望允许闭源商用，改用 MIT 但强制保留署名。
4. **后端逻辑随二进制分发**：Electron 主进程代码在 asar 包内，仍可被解包，但比明文 JS 难度高一个量级，已足够。
5. **水印可溯源**：导出文件水印建议含生成时间 + 用户标识（若将来接入用户系统），便于追责。
6. **不自动改 git config**：git author 修改需用户本人确认，本方案不在代码层强制。

## 11. 结论
对 PatentLens 而言，最务实有效的组合是：
**A 全套身份标注（永久署名 Alfred Shi）+ B1 发布混淆 + B2 核心 PROMPT/逻辑后移**。

这三项叠加后：
- 任何使用者都能在 UI、文件头、LICENSE 三处看到 Alfred Shi 是主贡献者；
- 直接拷走前端无法独立运行（缺 PROMPT、缺 AI 代理、缺 PDF 签发）；
- 前端 JS 即便被拿到也因混淆而难以二次维护；
- 不引入机器码/加壳等损害正版体验的强约束。

待用户确认范围后，再按第八节分期实施。
