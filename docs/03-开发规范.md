# 03 - 开发规范

## 1. Git 版本控制规范

### 1.1 提交信息格式

```
<type>(<scope>): <subject>
```

**type 类型**：

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复 Bug |
| docs | 文档更新 |
| style | 代码格式调整（不影响逻辑） |
| refactor | 重构（不新增功能、不修复 Bug） |
| test | 测试相关 |
| chore | 构建/工具/依赖变更 |

**scope 范围**（按模块）：

| 范围 | 说明 |
|------|------|
| core | 核心逻辑 |
| api | API 调用层 |
| ui | 前端界面 |
| patent-num | 专利号转换模块 |
| parser | 数据解析模块 |
| build | 构建配置 |

**示例**：
```
feat(api): 实现 USPTO ODP API 专利申请数据查询
fix(patent-num): 修复中国申请号校验位计算错误
docs: 补充 API 注册流程指南
refactor(core): 拆分审查历史解析逻辑
chore(build): 升级 Tauri 至 v2
```

### 1.2 分支策略

```
main          ← 稳定发布分支
├── dev       ← 开发主分支
├── feat/xxx  ← 功能分支
├── fix/xxx   ← 修复分支
└── docs/xxx  ← 文档分支
```

- 功能开发从 `dev` 拉取分支，完成后合并回 `dev`
- 发布版本从 `dev` 合并至 `main`
- 每次合并至 `main` 时打版本标签

### 1.3 强制规则

1. **每次提交后必须推送至远程仓库**
   ```bash
   git add .
   git commit -m "feat(api): add ODP API client"
   git push origin dev
   ```

2. **打包前必须校验代码一致性**
   ```bash
   git pull origin main
   git status          # 确认无未提交变更
   git diff origin/main  # 确认本地与远程一致
   ```

3. **禁止 force push 至 main 分支**

## 2. 密钥管理规范

### 2.1 .env 文件格式

在项目根目录创建 `.env` 文件：

```env
USPTO_API_KEY=your_api_key_here
```

### 2.2 强制规则

1. `.env` 文件**必须**加入 `.gitignore`，禁止提交至远程仓库
2. 项目根目录提供 `.env.example` 文件（不含真实密钥），供新成员参考
3. 代码中通过环境变量读取密钥，禁止硬编码
4. 日志中禁止输出完整 API Key（可脱敏显示前 4 位）

### 2.3 .env.example 模板

```env
USPTO_API_KEY=replace_with_your_api_key
```

## 3. 打包命名规范

### 3.1 命名格式

```
专利审查梳理工具_<版本号>_<系统平台>
```

### 3.2 示例

```
专利审查梳理工具_0.1.0_windows_x64.exe
专利审查梳理工具_0.1.0_linux_x64
```

### 3.3 版本号规则

- 遵循语义化版本（Semantic Versioning）：`MAJOR.MINOR.PATCH`
- 版本号必须对应 Git 中的版本标签
- 打包时自动从 Git tag 读取版本号

```bash
git tag v0.1.0
git push origin v0.1.0
```

## 4. 代码风格规范

### 4.1 Rust 代码风格

- 遵循 Rust 官方风格指南（rustfmt）
- 使用 `cargo fmt` 自动格式化
- 使用 `cargo clippy` 进行静态检查
- 命名规范：
  - 模块/文件名：`snake_case`
  - 类型/结构体/枚举：`PascalCase`
  - 函数/方法/变量：`snake_case`
  - 常量：`SCREAMING_SNAKE_CASE`

### 4.2 前端代码风格

- 缩进：2 空格
- 命名：
  - CSS 类名：`kebab-case`
  - JavaScript 变量/函数：`camelCase`
  - JavaScript 类：`PascalCase`

### 4.3 代码审查清单

- [ ] `cargo fmt` 无变更
- [ ] `cargo clippy` 无警告
- [ ] 无硬编码密钥或敏感信息
- [ ] API 调用间有适当间隔
- [ ] 错误处理完善（不使用 unwrap）
- [ ] 日志不包含完整 API Key

## 5. 依赖管理规范

### 5.1 版本锁定

- 使用 `Cargo.lock` 锁定所有依赖版本
- `Cargo.lock` **必须**提交至 Git
- 禁止随意升级依赖版本

### 5.2 新增依赖流程

1. 确认依赖的必要性和活跃度
2. 检查依赖的许可证兼容性
3. 指定精确版本号（不使用 `*` 或范围版本）
4. 提交时在 commit message 中说明新增依赖的原因

## 6. API 调用规范

### 6.1 请求频率

- 所有 USPTO API 调用之间至少等待 **1.5 秒**
- 实现请求队列，避免并发请求
- 遇 429 状态码时，指数退避重试（最多 3 次）

### 6.2 请求头

```rust
let client = reqwest::Client::builder()
    .user_agent("PatentHistoryHelper/0.1.0")
    .timeout(Duration::from_secs(30))
    .build()?;
```

### 6.3 数据使用

- 获取的专利文档**仅限内部梳理使用**
- **禁止**对外传播或商用
- 遵守 USPTO 数据使用条款
